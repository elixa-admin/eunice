# Parent-Led Next Three Sprints

Last updated: 2026-05-24
Primary surface: `dev/`
Branch: `codex/vercel-project-separation`

## Purpose

Build the next three sprints around the parent-led admissions journey, then carry the same cue language into Admin and the preview hub, while keeping the platform white-label ready for other schools later.

## Product Rules

- Primary school first in voice
- High-school heritage in visual polish
- Keep the experience simple, calm, and first-time-parent friendly
- Use cues, not clutter
- Preserve `dev/` as the active implementation surface
- Keep assessment untouched
- Keep `src/` untouched until explicit promotion
- Structure the platform so school branding and workflow rules can be configured later

## Sprint 1 - Parent Orientation And First-Time Confidence

### Phase 1.1 - Guided Start And Setup

Goal:
- Make the first screen feel like a welcoming admissions guide rather than a form wall.

What it should do:
- explain the journey in plain language
- state what to have ready
- set a simple time expectation
- explain why the key information matters
- reassure the parent they can save and return

Done when:
- a first-time parent understands what this is, why it matters, and how to begin without feeling overwhelmed

### Phase 1.2 - Simplified Before-You-Begin And Step Cueing

Goal:
- Replace multiple small prompts with one clear orientation panel and tighter step cues.

What it should do:
- consolidate the “before you begin” explanation
- keep the document purpose visible
- show the next step clearly
- make the screen feel lighter and more guided

Done when:
- the parent can scan the start of the application and quickly understand what is needed and why

## Sprint 2 - Document Confidence And Submission Readiness

### Phase 2.1 - Purpose-Led Document Readiness

Goal:
- Help parents understand documents by purpose instead of by a flat checklist.

What it should do:
- group documents into logical families
- explain why each one is needed
- clarify what is required now versus later
- reduce the sense of data overload

Done when:
- the parent can prepare confidently and complete uploads with less confusion

### Phase 2.2 - Review, Consent, And No-Surprises Submission

Goal:
- Make final review feel calm, safe, and predictable.

What it should do:
- surface what is complete, missing, or still reviewable
- explain what happens after submit
- make consent and final submission feel clear and trustworthy

Done when:
- the parent can review and submit without fearing hidden surprises

## Sprint 3 - Admin Operations And Whole-System Cue Cohesion

### Phase 3.1 - Admin Dashboard Barometers And Critical Items

Goal:
- Turn the Admin Dashboard into a proper operations view with obvious next actions.

What it should do:
- highlight blocking, review, and ready states
- make queue health obvious at a glance
- show what needs attention first
- keep the selected application easy to inspect

Done when:
- staff can scan the dashboard and know where to act immediately

### Phase 3.2 - Preview Hub And White-Label Architecture Readiness

Goal:
- Make the preview hub feel like a real control center and prepare the platform for future school branding.

What it should do:
- improve route cues and dashboard story
- align visual language across parent and admin surfaces
- document the school-config / theme-token architecture path for later white-label rollout

Done when:
- the `dev/` surface feels cohesive and the architecture is clearly ready for other schools later

## Shared Constraints

- Keep the assessment surface untouched
- Keep all active UI work on `dev/`
- Keep the brand primary-school appropriate
- Keep visuals restrained, clear, and functional
- Add complexity only when it improves clarity
